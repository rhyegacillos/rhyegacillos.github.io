Case Study · 03

Autonomous Agentic Trader

Production-Grade Multi-Agent Trading Infrastructure

Building a reliable trading system takes more than passing market data to an LLM (Large Language Model — the AI brain that reads text and generates responses, like GPT-4 or Gemini). This case study follows the full control loop: four named trader personas generate trade intent, a Redis-backed cycle cache gives them bounded market and research context, optional n8n workflows can produce deeper research, and a four-gate portfolio-control pipeline with 14 deterministic (always producing the same result for the same input — no randomness) risk rules decides what is allowed to execute. MCP (Model Context Protocol — a standard interface that controls which tools the AI can call and how) boundaries block direct market access, while SSE (Server-Sent Events — a technology that lets the server push live updates to the browser without the browser asking repeatedly) keeps the operator dashboard live through reconnects and resyncs.

Architecture Diagram

I

The Strategic Context: Why AI Trading Demos Fail in Production

Most "AI trading" demos share the same failure mode. The LLM gets direct access to a trading API. A system prompt tells it to "make good trades." The demo runs until it makes a bad one. There is no proposal lifecycle, no risk policy, no audit trail, and no operator override. Research is usually just another live tool call inside each agent run, which means four agents can duplicate the same search, read different context, or put pressure on upstream providers at exactly the moment the system needs stability. There is also no distinction between the LLM reasoning about trades and the system executing them. That is not a production architecture — it is a notebook experiment with a broker API key attached.

A production trading system requires a set of controls that demos consistently omit:

  • A controlled execution environment where every trade follows a deterministic (always producing the same result for the same input — no randomness) proposal → decision → execution lifecycle
  • A risk policy that checks proposals against portfolio state, not just price signals alone
  • A cycle-scoped research/cache layer that prevents four agents from multiplying the same provider calls
  • An operator layer that can inspect, approve, reject, or halt at any stage without stopping the engine
  • A persistence layer that makes every action recoverable and gives the ability to trace and review all actions after the fact

This project builds those controls as one continuous loop: prepare shared context, let agents propose, run deterministic portfolio-control, persist the decision trail, and keep the operator surface current.

II

The Four Trader Personas

The system runs four named LLM trader personas — Warren, George, Ray, and Cathie — each with a distinct risk profile and tuning parameters. At the start of a cycle, the app prepares shared price, news, event-risk, and research records so the personas are not independently scraping the same context before they reason. Their differences come from strategy and risk appetite, not from inconsistent tool access. These parameters are injected into both the agent instructions and the risk policy evaluation. They are not cosmetic. They feed directly into the risk gate's numerical thresholds, portfolio exposure cap multipliers, and re-entry timing logic. Each trader can also be independently configured to use a different LLM model — Gemini, OpenAI, DeepSeek, or Grok — via per-trader model routing.

Warren — Conservative

Warren defends existing positions first and is selective about new entries. He is skeptical of stale-price reactions and momentum-driven catalysts. Key parameters: rebuy_aggressiveness: 0.2 (how quickly to re-enter a position after exiting), loss_cut_strictness: 0.8 (how quickly to exit a losing position), new_entry_vs_defense_bias: 0.25 (how much to favour new trades over defending existing ones). Portfolio cap multiplier: 0.70×. Warren's instructions focus on capital preservation, high-conviction requirements for new positions, and patience over speed.

George — Aggressive

George moves fast on new entries and re-engages quickly after closing a position. He is catalyst-driven — he responds to news events and price momentum. Parameters: rebuy_aggressiveness: 0.7, loss_cut_strictness: 0.45, new_entry_vs_defense_bias: 0.75. Portfolio cap multiplier: 1.30×. George prioritises speed of entry, broad opportunity scanning, and willingness to absorb short-term losses for larger potential gains.

Ray — Balanced

Ray is aware of broad economic conditions and is mindful of risk. He leans toward defense when evidence is mixed or macro signals are unclear. He keeps balanced exposure across sectors. Parameters: rebuy_aggressiveness: 0.4, loss_cut_strictness: 0.6, new_entry_vs_defense_bias: 0.4. Portfolio cap multiplier: 1.00× (baseline). Ray focuses on cross-asset context, diversification, and explicit reasoning about economic conditions before entering new positions.

Cathie — Aggressive Growth

Cathie focuses on high-conviction growth opportunities. She prioritises signals of emerging technology adoption and long-term trend continuation over short-term valuation. Parameters: rebuy_aggressiveness: 0.8, loss_cut_strictness: 0.35, new_entry_vs_defense_bias: 0.8. Portfolio cap multiplier: 1.30×. Cathie looks for disruptive innovation, tolerates high volatility in fast-growing names, and concentrates capital in her highest-conviction positions.

III

The Portfolio-Control Pipeline: Four Gates

Shared research context does not give traders execution authority. No trader agent executes a trade directly. Every buy_shares or sell_shares MCP tool call routes to proposal intake, not to the Alpaca execution API. The tool name matches the trader's mental model — it expresses what the trader wants to do. But the implementation sends that call into a controlled pipeline instead. The proposal then travels through four deterministic gates before anything touches a real brokerage position.

Gate 1 — Proposal Intake

The system normalises, persists, and deduplicates proposals by hash of (agent_name, symbol, side, quantity, time_bucket) using 5-minute time buckets. This stops the same trader from submitting identical proposals within the same cycle window. Proposals expire after 30 minutes via TTL (Time-To-Live — an automatic expiry timer that deletes data after a set period) enforcement. Cycle-based duplicate blocking also prevents the same symbol/side combination from being proposed twice in one cycle, even from different traders. Status progression: created → validated → decided (or rejected_invalid / expired / duplicate).

Gate 2 — Deterministic Risk Policy (14+ Rules)

The risk policy checks each validated proposal against a fresh portfolio snapshot pulled at evaluation time. The Redis cycle cache can reduce upstream provider work, but portfolio-control still anchors decision authority to the current-cycle quote snapshot and the persisted portfolio state. Rules run in sequence:

  • Portfolio state freshness check (reject if snapshot is stale beyond threshold)
  • Preflight receipt validation (reject without valid preflight receipt)
  • Sell position validation (must hold shares to sell; oversized sells are rejected or reduced)
  • Drawdown entry block (block new entries if portfolio is down more than 10%)
  • Drawdown exposure reduction (reduce existing exposure if drawdown deepens)
  • Position exposure limits (maximum 10% per symbol of total portfolio value)
  • Deployed capital limits (cap total deployed portfolio exposure)
  • Confidence threshold (reject low-confidence proposals)
  • Averaging-down block (prevent adding to a losing position)
  • Re-entry cooldown (enforce a waiting period after a recent exit)
  • Minimum actionable size ($250–$400 notional minimum)
  • Intraday limits (frequency and daily volume caps)
  • Price source adequacy check (reject on unreliable or low-trust pricing)
  • Offsetting signal detection (blocks a buy when an open sell exists for the same symbol)

Each rule produces a typed reason code. Outcomes: approved, reduced (quantity reduced to fit within limits with an explicit reduction reason), or rejected with the specific rule and reason code attached.

Gate 3 — Execution Gateway

Three execution modes decide what happens to an approved proposal. proposal_only: the execution path is blocked entirely. Proposals are captured for review but never submitted to the broker. approval_required: proposals wait in a queue for manual operator approval via the dashboard, with TTL expiry if not acted on. auto_execute: approved proposals proceed to Alpaca automatically. In approval_required mode, the system runs revalidation checks before submitting to the broker — pricing must still be fresh and the risk policy must still pass against current portfolio state. A proposal approved 10 minutes ago may no longer be valid if the position changed. Full status chain: pending_approval → approved → submitted → filled (or failed / rejected_by_human / expired). All execution processing is asynchronous (running in the background without blocking other tasks). A background ExecutionWorker daemon thread polls the execution_work_items queue and processes each execution through revalidation, broker submission, reconciliation, and final state recording with retry and backoff. In auto_execute mode, approved decisions still persist first as pending_approval, then the system immediately creates work items that move them into the async queue without waiting for a human click. Non-terminal executions are re-enqueued on startup for recovery.

Gate 4 — Supporting Review / Extra Review (Optional)

This gate triggers when two or more proposals conflict on the same symbol. Three conflict types are handled: a same_symbol_opposite_side conflict (one trader buying while another is selling), a same_symbol_same_side_competing conflict (multiple traders competing for the same entry), or a same-sector competing allocation conflict. The system routes conflicting proposals to an advisory LLM for analysis. The LLM considers each trader's persona parameters and current portfolio context. Deterministic policy always has final authority — the LLM output is a recommendation, not a decision. A shadow-style review path is available for running the analysis in parallel without blocking the primary pipeline.

IV

MCP, Cache, and Research Boundaries

The previous chapters describe who proposes and who decides. This boundary layer explains how agents gather context without bypassing either side of that split. Traders never call market APIs, account APIs, or research workflows directly. All access flows through typed MCP tool contracts and cache-aware service boundaries. This makes every external interaction traceable at the tool boundary, independently testable, and replaceable without modifying the agent loop. A failed MCP session for one trader is isolated and retried independently. It does not bring down the sessions of other traders running in the same cycle.

Accounts Server

Tools: get_balance, get_holdings, buy_shares, sell_shares, change_strategy. The buy_shares and sell_shares tools are named as if they execute directly. The trader calls what it intends to do. But the implementation submits a structured proposal to the four-gate pipeline instead. This keeps the trader's reasoning natural while enforcing full pipeline control on the execution side.

Market Server

Tools: lookup_share_price, get_quote_freshness, get_symbol_news_summary, get_symbol_event_risk, get_symbol_research_brief. During each scheduler cycle, the app pre-warms a Redis cache keyed by {kind}:{cycle_id}:{symbol} for price, news, event-risk, and research records. Trader-side market tools read that cycle cache first, then call an internal FastAPI cache endpoint on miss, and only then trigger upstream provider work. This keeps four concurrent personas from making four copies of the same Alpaca, Polygon, Brave, Playwright, or research workflow call. Portfolio-control decision and execution authority still uses strict current-cycle quote snapshots fed by the Alpaca → Polygon path — prior-cycle Redis or persisted quote records are never treated as fresh authority for a new decision.

Researcher and n8n Boundary

Research production is switchable. In internal mode, the app derives research briefs during pre-warm from its own news/event pipeline and can fall back to a Researcher sub-agent with Web Tools MCP servers such as Playwright, Brave Search, and Fetch. In n8n mode, FastAPI keeps lifecycle ownership but delegates heavy source fan-out, page reading, summarization, filtering, and retry behavior to a configured n8n workflow. The workflow writes back to the same Redis cache contract, so trader prompts and MCP tools do not need to know which producer generated the research brief. If the cached brief is sufficient, the trader never invokes deeper research at all.

Memory Server (PostgreSQL Knowledge Graph)

Per-trader knowledge graph stored in PostgreSQL — create_entities, create_relations, search_nodes, read_graph. Traders build up research context across cycles: companies analysed, market conditions observed, sector dynamics, prior trade outcomes, and analyst notes. This forms a persistent memory that informs future reasoning without requiring the operator to re-inject context at the start of each cycle. Memory is scoped per trader — Warren's knowledge graph does not mix with Cathie's.

V

Real-Time Architecture: SSE with Replay and Resync

The operator dashboard must stay live across reconnects, engine restarts, and network interruptions without losing state or showing stale data. A simple WebSocket or polling endpoint is not enough. The system needs a delivery mechanism that replays missed messages on reconnect, falls back gracefully when the stream is unavailable, and stays consistent whether the engine is running or stopped.

SSE Broker with 300-Event Ring Buffer

The dashboard receives all state updates via a single SSE stream. The SSE broker maintains a ring buffer (a fixed-size memory store that keeps the most recent N items, overwriting the oldest when full) of 300 events keyed by sequence ID. On reconnect, the client sends Last-Event-ID and the broker replays all missed events from that position forward. The client rebuilds any state changes that occurred during the disconnection without a full page reload. The broker keeps a per-subscriber queue of 64 items and uses sequential event IDs in evt_YYYYMMDD_NNNNNN format. The stream sends heartbeat comments every 20 seconds to keep connections alive through proxies. If the Last-Event-ID falls outside buffer coverage — meaning the disconnection was longer than the buffer window — the client detects the desync condition and triggers a full REST resync to rebuild current state before resuming stream consumption.

Disconnected Fallback Polling

When the SSE stream is unavailable — network interruption, server restart, connection drop — the frontend switches to polling the versions endpoint every 20 seconds to detect state changes that need a UI update. Polling stops the moment the stream reconnects successfully. The dashboard falls back to polling under bad network conditions without requiring operator intervention. It immediately restores the more efficient SSE path once connectivity is re-established.

Event Types and Surface Refresh Discipline

The SSE stream carries 12 distinct event types across 6 scopes — bootstrap, manager, trader, scheduler, market, and execution_mode. Events include bootstrap_updated, manager_updated, proposal_created, decision_updated, execution_updated, trader_updated, trader_logs_updated, scheduler_updated, market_updated, market_status_changed, execution_mode_updated, and resync_required.

Surface refresh is conditional by design. Active surfaces refresh immediately on relevant events via non-blocking background fetches. Inactive surfaces are marked stale and refresh only when activated. During bursts of related events, coalescing prevents duplicate overlapping fetches.

Read-Model Layer for Dashboard Consistency

The dashboard query paths use a dedicated read-model with two operating modes. Persisted shells: when the trading engine is stopped, the dashboard is served from cached read-model snapshots stored in PostgreSQL. The operator sees correct portfolio state even with the engine offline. Live reads: when the engine is running, queries go directly against current state for correctness under rapidly changing position data. There is no blank or stale state window during engine restarts, between cycles, or during any stop/start transition.

VI

Operations: Persistence, Deployment, Operator Controls

FastAPI (Python 3.12)

Control plane, static file host, and SSE broker. Serves both the API and the static Next.js frontend on port 8000. 40+ REST endpoints covering scheduler lifecycle, proposal management, execution controls, read-model queries, and audit log access.

ExecutionWorker (daemon thread)

Background daemon thread that polls the execution_work_items queue and processes executions asynchronously. Starts on app startup, stops on shutdown. Handles approval-time revalidation, broker mutation, reconciliation, and terminal state emission with retry/backoff.

Market-Status Watcher

Background watcher that polls market status every 60 seconds. Drives the backend-controlled scheduler auto-start at market open — previously browser-driven, now handled server-side so the scheduler can start without an active browser tab.

Next.js 15 / React 19 (static export)

Dashboard built to static HTML/JS, served directly by FastAPI. No separate frontend server required — the entire frontend is a deployment artifact bundled into the container image. The operator workspace provides 5 tabs: Manager overview and per-Trader views for Warren, George, Ray, and Cathie.

PostgreSQL 16 + SQLAlchemy 2.x + Alembic

Source of truth. 30+ tables covering proposals, decisions, executions, audit events, position lots, closed trades, performance snapshots, memory graphs, read models, and supporting-review/adjudication records. Alembic migrations applied automatically at container startup.

OpenAI Agents SDK

Trader agent orchestration with bounded multi-turn runs (MAX_TURNS=30) and MCP session management. Turn limits prevent runaway agent loops from consuming unbounded compute.

MCP (stdio servers)

Trader-facing Accounts, Market, and Memory servers, plus Web Tools MCP for the Researcher sub-agent. Failed MCP sessions are isolated per-trader and retried independently. All tool calls are logged at the MCP boundary for full auditability.

Alpaca + Polygon + Brave Search

Alpaca handles trade execution and account state after full four-gate approval and is the primary strict quote provider. Polygon is the structured price fallback. Brave Search is research/news only and is no longer part of price authority for manager decisions or execution approval.

Redis / Upstash Cycle Cache

Cycle-scoped cache for price, news, event-risk, and research records. Local Docker Redis is used in development; Upstash REST is supported in production. Cache keys include the active cycle ID, so old-cycle data cannot satisfy a new cycle's research or quote lookup by accident.

n8n Research Workflows

Optional external producer for pre-warm research and on-demand deeper research. FastAPI posts the cycle payload to n8n, waits for Redis readiness status, and keeps the proposal, decision, execution, and audit lifecycle inside the app.

Cycle-Scoped Quote Authority

Each decision, preflight, or execution cycle starts with a fresh in-memory quote cache. Symbols are fetched once per cycle and reused within that cycle, but prior-cycle cache and persisted quote records are never authoritative for a new cycle — preventing stale prices from leaking into live decisions while still avoiding rate-limit pressure from repeated same-cycle lookups.

Execution Recovery

On startup, non-terminal executions are detected and re-enqueued to the ExecutionWorker queue. In auto_execute mode, recovered items are automatically re-enqueued so the execution pipeline resumes without operator intervention after process restarts.

Docker + AWS EC2/ECR

Single containerized deployment on EC2. Chosen over serverless because the trading engine runs as a long-lived subprocess — Lambda cold starts would break the persistent MCP server sessions, the scheduler lifecycle, and the SSE broker state.

Nginx + Certbot (HTTPS termination)

Optional Nginx reverse proxy with Certbot-managed TLS certificates terminates HTTPS on the EC2 instance. Configured with extended upstream timeouts and structured access logging including request path, status, timing, and upstream response time.

Terraform

Infrastructure as code for EC2, RDS, ECR, Secrets Manager, and Route 53. All infrastructure is reproducible from source — no manual console configuration. Shared modules with per-environment variable overrides for staging/production parity.

GitHub Actions

CI/CD pipeline: build Docker image, push to ECR, deploy to EC2 via SSM. No exposed SSH ports and no long-lived access credentials in the pipeline. Triggered on push to main and available for manual hotfix deploys.

VII

Architectural Result

The final shape is not "agents that trade." It is a controlled trading simulation where agents create structured intent, shared context is prepared before they run, optional n8n workflows can expand research production, and deterministic portfolio-control decides whether anything is allowed to move forward. The external workflow layer improves research throughput; it does not become the trading authority.

That separation is the main engineering point of the project. Trader reasoning, research production, risk policy, execution, persistence, and dashboard state all have clear boundaries. The system can add richer research or different model providers without weakening the proposal → decision → execution contract that keeps the operator in control.
