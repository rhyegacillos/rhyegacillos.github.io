Autonomous Agentic Trader

Production-Grade Multi-Agent Trading Infrastructure

Building a trading system that separates signal from noise requires more than calling an LLM with market data. This case study covers the four-gate portfolio-control pipeline with 14 deterministic risk rules, four named trader personas with individually tuned behavioral parameters, the MCP tool boundaries that prevent direct market access, and the SSE-first real-time architecture that keeps the operator dashboard live under reconnect and resync scenarios.

Architecture Diagram

I

The Strategic Context: Why AI Trading Demos Fail in Production

Most "AI trading" demos share the same failure mode: the LLM has direct access to a trading API, a system prompt tells it to "make good trades," and the demo runs until it makes a bad one. There is no proposal lifecycle, no risk policy, no audit trail, no operator override, and no distinction between the LLM reasoning about trades and the system executing them. This is not a production architecture — it is a notebook experiment that happens to have a broker API key attached.

A production trading system requires four things that demos consistently omit:

  • A controlled execution environment where every trade follows a deterministic proposal → decision → execution lifecycle
  • A risk policy that evaluates proposals against portfolio state, not just price signals in isolation
  • An operator layer that can inspect, approve, reject, or halt at any stage without stopping the engine
  • A persistence layer that makes every action recoverable and auditable after the fact

This project builds all four as first-class systems.

II

The Four Trader Personas

The system runs four named LLM trader personas — Warren, George, Ray, and Cathie — each with a distinct risk profile and tuning parameters injected into both their agent instructions and the risk policy evaluation. These parameters are not cosmetic — they feed directly into the risk gate's numerical thresholds, portfolio exposure cap multipliers, and re-entry timing logic. Each trader can also be independently configured to use a different LLM model — Gemini, OpenAI, DeepSeek, or Grok — via per-trader model routing.

Warren — Conservative

Defensive first. Selective on new entries. Skeptical of stale-price reactions and momentum-driven catalysts. Key parameters: rebuy_aggressiveness: 0.2, loss_cut_strictness: 0.8, new_entry_vs_defense_bias: 0.25. Portfolio cap multiplier: 0.70×. Warren's instructions emphasize capital preservation, high conviction requirements for new positions, and patience over reactivity.

George — Aggressive

Aggressive on fresh entries. Faster to re-engage after a position is closed. Catalyst-driven — responds quickly to news events and price momentum. Parameters: rebuy_aggressiveness: 0.7, loss_cut_strictness: 0.45, new_entry_vs_defense_bias: 0.75. Portfolio cap multiplier: 1.30×. Emphasizes speed of entry, broad opportunity scanning, and willingness to absorb short-term drawdown for asymmetric upside.

Ray — Balanced

Macro-aware and risk-conscious. Defense bias when evidence turns mixed or macro signals are ambiguous. Maintains balanced exposure across sectors. Parameters: rebuy_aggressiveness: 0.4, new_entry_vs_defense_bias: 0.4. Portfolio cap multiplier: 1.00× (baseline). Emphasizes cross-asset context, diversification discipline, and explicit reasoning about macro conditions before entering new positions.

Cathie — Aggressive Growth

High-conviction growth focus. Innovation catalyst-first — prioritizes emerging technology adoption signals and long-duration trend continuation over near-term valuation. Parameters: rebuy_aggressiveness: 0.8, loss_cut_strictness: 0.35, new_entry_vs_defense_bias: 0.8. Portfolio cap multiplier: 1.30×. Emphasizes disruptive innovation thesis validation, tolerance for volatility in high-growth names, and concentration in high-conviction positions.

III

The Portfolio-Control Pipeline: Four Gates

No trader agent executes a trade directly. Every buy_shares or sell_shares MCP tool call routes to proposal intake, not to the Alpaca execution API. The tool is intentionally named to match the trader's mental model — it calls what it wants to happen — but the implementation routes that call into a controlled pipeline. The proposal then travels through four deterministic gates before anything touches a real brokerage position.

Gate 1 — Proposal Intake

Proposals are normalized, persisted, and deduplicated by hash of (agent_name, symbol, side, quantity, time_bucket) with 5-minute time buckets — preventing the same trader from submitting economically identical proposals within the same cycle window. Proposals expire after 30 minutes via TTL enforcement. Cycle-based duplicate blocking prevents the same symbol/side combination from being proposed twice in one cycle, even from different traders. Status progression: created → validated → decided (or rejected_invalid / expired / duplicate).

Gate 2 — Deterministic Risk Policy (14+ Rules)

The risk policy evaluates each validated proposal against a fresh portfolio snapshot pulled at evaluation time. Rules applied in sequence:

  • Portfolio state freshness check (reject if snapshot is stale beyond threshold)
  • Preflight receipt validation (reject without valid preflight receipt)
  • Sell position validation (must hold shares to sell; oversized sells are rejected or reduced)
  • Drawdown entry block (block new entries if portfolio is down more than 10%)
  • Drawdown exposure reduction (reduce existing exposure if drawdown deepens)
  • Position exposure limits (maximum 10% per symbol of total portfolio value)
  • Deployed capital limits (cap total deployed portfolio exposure)
  • Confidence threshold (reject low-confidence proposals)
  • Averaging-down block (prevent adding to a losing position)
  • Re-entry cooldown (enforce a waiting period after a recent exit)
  • Minimum actionable size ($250–$400 notional minimum)
  • Intraday limits (frequency and daily volume caps)
  • Price source adequacy check (reject on unreliable or low-trust pricing)
  • Offsetting signal detection (blocks a buy when an open sell exists for the same symbol)

Each rule produces a typed reason code. Outcomes: approved, reduced (quantity reduced to fit within limits with an explicit reduction reason), or rejected with the specific rule and reason code attached.

Gate 3 — Execution Gateway

Three execution modes determine what happens to an approved proposal: proposal_only (the execution path is blocked entirely; proposals are captured for review but never submitted to the broker), approval_required (proposals are queued for manual operator approval via the dashboard, with TTL expiry if not acted on), auto_execute (approved proposals proceed to Alpaca automatically). On operator approval in approval_required mode, revalidation checks run before submission — pricing must still be fresh and the risk policy must still pass against current portfolio state. A proposal approved 10 minutes ago may no longer be valid if the position changed. Full status chain: pending_approval → approved → submitted → filled (or failed / rejected_by_human / expired).

Gate 4 — Supporting Review / Extra Review (Optional)

Triggered when 2 or more proposals conflict on the same symbol: a same_symbol_opposite_side conflict (one trader buying while another is selling), a same_symbol_same_side_competing conflict (multiple traders competing for the same entry), or a same-sector competing allocation conflict. Routes to an advisory LLM for supporting review or extra review analysis that considers each trader's persona parameters and current portfolio context. Deterministic policy always has final authority — the LLM output is a recommendation, not a decision. A shadow-style supporting review path is available for running the analysis in parallel without blocking the primary pipeline path.

IV

MCP Tool Boundaries: The Isolation Contract

Traders never call market APIs or account APIs directly — all access flows through typed MCP tool contracts. This makes every external interaction auditable at the tool boundary, independently testable, and replaceable without modifying the agent loop. A failed MCP session for one trader is isolated and retried independently — it does not collapse the sessions of other traders running in the same cycle.

Accounts Server

Tools: get_balance, get_holdings, buy_shares, sell_shares, change_strategy. The buy_shares and sell_shares tools are intentionally named as if they execute — the trader calls what it semantically intends — but the implementation submits a structured proposal to the four-gate pipeline. This keeps the trader's reasoning naturalistic while enforcing full pipeline control on the execution side.

Market Server

Tools: lookup_share_price (multi-source resolution chain: Alpaca → Polygon → cache), get_quote_freshness, get_symbol_news_summary, get_symbol_event_risk. Price source quality is tracked across the resolution chain — stale or low-trust quotes trigger the quote_low_trust routing path in Gate 2, which escalates to the price source adequacy rule and may reject the proposal rather than execute on unreliable pricing.

Memory Server (PostgreSQL Knowledge Graph)

Per-trader knowledge graph stored in PostgreSQL — create_entities, create_relations, search_nodes, read_graph. Traders accumulate research context across cycles: companies analyzed, market conditions observed, sector dynamics, prior trade outcomes, and analyst notes. This forms a persistent memory that informs future reasoning without requiring the operator to re-inject context at the start of each cycle. Memory is scoped per trader — Warren's knowledge graph does not contaminate Cathie's.

V

Real-Time Architecture: SSE with Replay and Resync

The operator dashboard needs to stay live across reconnects, engine restarts, and network interruptions without losing state or showing stale data. This requires more than a simple WebSocket or polling endpoint — it requires a delivery mechanism that handles message replay on reconnect, graceful fallback when the stream is unavailable, and a consistency model that is correct whether the engine is running or stopped.

SSE Broker with 300-Event Ring Buffer

The dashboard receives all state updates via a single Server-Sent Events stream. The SSE broker maintains a 300-event ring buffer keyed by event sequence ID. On reconnect, the client sends Last-Event-ID and the broker replays all missed events from the buffer position forward — the client reconstructs any state changes that occurred during the disconnection without a full page reload. If the Last-Event-ID falls outside buffer coverage (disconnection longer than the buffer window), the client detects the desync condition and triggers a full REST resync to rebuild current state before resuming stream consumption.

Disconnected Fallback Polling

When the SSE stream is unavailable — network interruption, server restart, connection drop — the frontend switches to polling the versions endpoint every 20 seconds to detect state changes that require a UI update. Polling stops the moment the stream successfully reconnects. The dashboard degrades gracefully to polling under adverse network conditions without requiring operator intervention, and immediately restores the more efficient SSE path once connectivity is re-established.

Read-Model Layer for Dashboard Consistency

The dashboard query paths use a dedicated read-model with two operating modes: persisted shells (when the trading engine is stopped, the dashboard is served from cached read-model snapshots persisted in PostgreSQL — the operator sees correct portfolio state even with the engine offline) and live reads (when the engine is running, queries are made directly against current state for correctness under rapidly changing position data). There is no blank or stale state window during engine restarts, between cycles, or under any stop/start transition.

VI

Operations: Persistence, Deployment, Operator Controls

FastAPI (Python 3.12)

Control plane, static file host, and SSE broker. Serves both the API and the static Next.js frontend on port 8000. 40+ REST endpoints covering scheduler lifecycle, proposal management, execution controls, read-model queries, and audit log access.

Next.js 14 (static export)

Dashboard built to static HTML/JS, served directly by FastAPI. No separate frontend server required — the entire frontend is a deployment artifact bundled into the container image.

PostgreSQL 16 + SQLAlchemy 2.x + Alembic

Source of truth. 30+ tables covering proposals, decisions, executions, audit events, position lots, closed trades, performance snapshots, memory graphs, read models, and supporting-review/adjudication records. Alembic migrations applied automatically at container startup.

OpenAI Agents SDK

Trader agent orchestration with bounded multi-turn runs (MAX_TURNS=30) and MCP session management. Turn limits prevent runaway agent loops from consuming unbounded compute.

MCP (stdio servers)

Three servers: Accounts, Market, and Memory. Failed MCP sessions are isolated per-trader and retried independently. All tool calls are logged at the MCP boundary for full auditability.

Alpaca + Polygon + Brave Search

Alpaca for trade execution and account state — accessed only through the Accounts MCP server after full four-gate approval. Alpaca and Polygon handle market price resolution, while Brave Search is used for research, news summaries, and event-risk context.

Docker + AWS EC2/ECR

Single containerized deployment on EC2. Chosen over serverless because the trading engine runs as a long-lived subprocess — Lambda cold starts would break the persistent MCP server sessions, the scheduler lifecycle, and the SSE broker state.

Terraform

Infrastructure as code for EC2, RDS, ECR, Secrets Manager, and Route 53. All infrastructure is reproducible from source — no manual console configuration. Shared modules with per-environment variable overrides for staging/production parity.

GitHub Actions

CI/CD pipeline: build Docker image, push to ECR, deploy to EC2 via SSM. No exposed SSH ports and no long-lived access credentials in the pipeline. Triggered on push to main and available for manual hotfix deploys.
