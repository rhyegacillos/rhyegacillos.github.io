Case Study · 04

MediNotes

Agentic Clinical Documentation with Multi-Agent QA

Clinical documentation has a correctness imperative that most AI systems ignore. This case study covers the hub-and-spoke multi-agent pipeline that separates extraction from synthesis from critique, the Critic QA loop that catches hallucinations before they reach the clinician, the Evidence Linking protocol that maps every summary claim back to its source, and the persistent format guardrails that improve output quality across sessions without manual prompt tuning.

Architecture Diagram

I

The Strategic Context: Correctness as a First-Class Requirement

Clinical documentation operates in a domain where "good enough" is not acceptable. An LLM that hallucinates a drug interaction, omits a contraindication, or invents a dosage instruction is not producing a suboptimal output — it is creating a patient safety risk. Most AI summarization systems treat the LLM as a single-step black box: send all the context in, receive a summary out. This architecture fails for clinical use because it provides no mechanism to detect or correct errors before the output reaches the clinician.

The correctness requirements for MediNotes were defined at the outset:

  • Every claim in the summary must be traceable to a source document
  • Hallucinations must be caught before delivery, not after
  • Multimodal inputs must be handled without information loss during format conversion
  • The pipeline must be auditable at each stage — not just at the final output

These requirements ruled out a single-prompt architecture from the start and drove the hub-and-spoke agent topology.

II

The Hub-and-Spoke Agent Topology

Rather than a single monolithic prompt that handles extraction, synthesis, research, critique, and memory in one pass, the pipeline uses a hub-and-spoke topology where the Summary Agent orchestrates specialized agents, each with a single clearly bounded responsibility. A failure in the Research Agent does not contaminate the Extraction Agent's output, and a Critic failure triggers a targeted regeneration cycle — not a full pipeline restart.

Extraction Agent — Multimodal Input Normalization

Converts all input formats to a unified text representation before any summarization occurs. OCR via vision model for prescription images and scanned documents where a text layer is not available. OpenAI Whisper transcription for audio recordings of consultation notes. Text extraction for PDFs with embedded text layers and DOCX documents. By normalizing inputs in a dedicated extraction stage, the Summary Agent always works with clean, uniform context — it never receives raw binary files, half-parsed PDFs, or unprocessed audio blobs.

Research Agent — Conditional External Lookup

Conditionally triggered — not on every run, but when the visit context contains signals that warrant external verification: potential drug interactions, unfamiliar medication names, dosage ranges outside common clinical norms, or guidelines requiring current evidence. Queries external clinical sources via the Brave Search MCP server. Results are injected as clearly marked referenced external evidence into the Summary Agent's generation step — not as direct clinical claims. This prevents the research step from introducing new hallucinations while still enriching the summary with relevant external context.

Summary Agent — Orchestration Hub

Builds a unified visit context from all extracted inputs and any research evidence gathered by the Research Agent. Composes the structured 3-section HTML summary: chief complaint and assessment, clinical findings and observations, treatment plan and follow-up. Acts as the hub that sequences and assembles pipeline outputs — coordinating deterministic handoffs to the Critic Agent for QA and the Evidence Agent for source mapping. Orchestration logic is explicit: handoffs are deterministic, not emergent from an open-ended agent loop.

Critic Agent — Pre-Delivery QA Gate

Evaluates every generated summary before it reaches the clinician. The Critic Agent checks for four failure categories:

  • Factual contradictions — claims that directly conflict with the extracted source input
  • Clinical omissions — findings present in the source that are absent from the summary
  • Hallucinated specifics — names, dosages, dates, or measurements with no source support
  • Structural violations — deviations from the expected 3-section format

If the summary fails on any category, the system runs a tournament of up to 3 regenerated candidates with the Critic's failure analysis injected as negative constraints, and selects the best-scoring output. This adds latency but eliminates the delivery of unverified clinical claims to the clinician.

III

The Integrity Layer: Evidence Linking & Format Guardrails

The Critic Agent catches hallucinations before delivery. Evidence Linking makes every surviving claim auditable after delivery. Together with Persistent Format Guardrails, these three mechanisms implement a layered correctness contract — no single mechanism is sufficient for clinical-grade output quality, and all three work together to close different categories of error.

Evidence Agent — Source-to-Claim Mapping

Every statement in the finalized summary is mapped back to the specific source chunk that supports it. The Evidence Agent processes the finalized summary and the original extraction output, producing a claim-to-source index that the clinician can inspect in the UI — any claim can be expanded to show the exact input excerpt it was derived from. In a clinical context, auditability is a safety requirement. If a statement cannot be traced to a source, it is flagged as unsupported and surfaced as an evidence gap — not silently included in the final output.

Persistent Format Guardrails

When the Critic Agent identifies format or structural violations in a summary — missing section headers, incorrect subsection ordering, malformed HTML structure, clinical information placed in the wrong section — those violations are persisted as format guardrails for the current doctor's profile in the database. Future generation prompts for this doctor include the accumulated guardrails as explicit negative constraints injected into the system prompt. Output quality improves across sessions without any manual prompt engineering: the pipeline learns from its own Critic failures and encodes that learning as persistent, doctor-scoped generation constraints.

Zero Hallucination Contract

The combination of Critic QA, Evidence Linking, and Persistent Guardrails implements a layered zero hallucination contract. The Critic catches errors in the current run. Evidence Linking makes any errors that survive the Critic visible to the clinician before they act on the summary. Persistent Guardrails reduce the probability of the same class of errors appearing in future runs. No single layer is sufficient; the contract requires all three working in sequence.

IV

Longitudinal Memory: RAG for Patient History

A clinician working with a returning patient should not need to re-provide history that is already documented. The longitudinal memory layer makes prior visit context automatically available to the Summary Agent without requiring the clinician to manually locate and re-attach previous summaries, and without the privacy risk of injecting unscoped history from other patients.

DynamoDB for Cross-Visit Context

After each visit, the finalized summary is embedded and stored in a DynamoDB table keyed by patient ID. On subsequent visits for the same patient, semantically similar context from prior summaries is retrieved via similarity search and injected as background context into the Summary Agent's generation step — prior diagnoses, known allergies, current medication history, and documented clinical observations. Retrieval is scoped strictly to the current patient's indexed summaries — cross-patient retrieval is architecturally impossible because the DynamoDB queries are keyed by patient ID at the query layer.

Scoped Chat Assistant

The Chat Agent gives the clinician a conversational interface for asking questions about the current patient's history and the current visit context. It has read access only to the current patient's visit summaries and consultation context. Cross-patient data access is architecturally impossible, not just policy-constrained: the chat session is initialized with a patient-scoped context object that is the only data source available to the agent. Clinical answers are always grounded in documented evidence — the Chat Agent cannot speculate beyond what is in the patient's record.

V

Real-Time Pipeline Streaming

The full pipeline — extraction, optional research, summarization, Critic QA, evidence linking, and memory indexing — takes 30 to 60 seconds under typical load. Blocking the UI until completion is not acceptable: the clinician has no visibility into what the pipeline is doing, and loses all work if connectivity drops during the wait.

SSE-Driven Incremental Updates

Each pipeline stage emits an SSE event as it finishes: extraction_complete, research_done, summary_draft_ready, critic_pass_complete, evidence_linked, memory_indexed. The frontend renders incrementally as stages complete, giving the clinician real-time visibility into pipeline progress. If the Critic triggers a regeneration tournament, a critic_regenerating event is emitted so the clinician knows why the pipeline is taking longer. The UI is never in an ambiguous pending state.

Resumable SSE Jobs

Job state is persisted in Upstash Redis keyed by job_id. If the client disconnects mid-pipeline — browser tab closed, network interruption, mobile app backgrounded — the pipeline continues running on the server. The client reconnects with the job_id and resumes streaming from the last completed stage event. No work is re-executed; no stage outputs are lost. The Redis job record tracks the completion state of each pipeline stage, so the resumption point is exact — the client receives only the events it missed, not a full replay from the beginning.

VI

Operations

FastAPI (Python)

Pipeline orchestration, SSE streaming host, multimodal file handling, and job state management. Handles agent handoff coordination between pipeline stages and exposes the SSE endpoint for incremental progress updates.

Next.js / React + TypeScript

SSE event consumption and incremental stage rendering — each stage event triggers a UI update. Evidence linking UI lets clinicians expand any summary claim to inspect its source excerpt. Scoped chat interface per patient session.

OpenAI Agents SDK

Hub-and-spoke orchestration. Handoffs between Summary, Extraction, Research, Critic, and Evidence agents via structured handoff primitives. Bounded agent runs prevent open-ended loops in the Critic regeneration tournament.

OpenAI Whisper

Audio transcription for voice-recorded consultation notes. Runs within the Extraction Agent before any summarization; output normalized to the same text format as all other inputs so the Summary Agent sees a uniform context.

DynamoDB (RAG)

Longitudinal patient memory via embedding and similarity retrieval. Summaries indexed by patient ID. Cross-patient retrieval is architecturally prevented at the query layer — not just by policy.

Upstash Redis

Job state persistence for resumable SSE sessions. Each pipeline job stored as a Redis hash keyed by job_id with per-stage completion flags. HTTP-based access compatible with App Runner without persistent TCP connections.

MCP (Brave Search)

Research Agent's external lookup mechanism for clinical guidelines and drug interaction data. All external queries go through the typed tool contract; tool call logs captured per pipeline run for auditability.

Resend

Email delivery for summary dispatch. Verified-send pattern: server confirms send receipt before recording delivery success. Failed sends surface as delivery errors in the job state — not silent failures.

Clerk

Authentication and user identity. Scopes all patient data access by authenticated user. JWTs verified on every API request. Patient DynamoDB queries and job history queries both scoped to the verified user identity before any data operation.

Docker + AWS App Runner + ECR

Containerized deployment with automatic HTTPS, health check routing, and request-based scaling. App Runner provides the persistent compute environment needed for multi-stage pipeline jobs that exceed Lambda timeout limits.
