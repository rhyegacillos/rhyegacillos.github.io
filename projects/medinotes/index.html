Case Study · 04

MediNotes

Agentic Clinical Documentation with Multi-Agent QA

Clinical documentation requires correctness — a standard most AI systems do not meet. This case study covers the hub-and-spoke multi-agent pipeline that separates extraction, synthesis, and critique into distinct stages, the Critic QA loop that catches hallucinations before they reach the clinician, the Evidence Linking protocol that maps every summary claim back to its source document, and the persistent format guardrails that improve output quality across sessions without any manual prompt tuning.

Architecture Diagram

I

The Strategic Context: Correctness as a First-Class Requirement

Clinical documentation does not allow for "good enough." An LLM (Large Language Model — the AI brain that reads text and generates responses, like GPT-4 or Gemini) that invents a drug interaction, misses a contraindication, or creates a dosage instruction that was not in the source material is not producing a bad output — it is creating a patient safety risk. Most AI summarisation systems treat the LLM as a single step: send all the context in, receive a summary out. This fails for clinical use because there is no way to catch or fix errors before the output reaches the clinician.

The correctness requirements for MediNotes were defined from the start:

  • Every claim in the summary must be traceable to a source document
  • Hallucinations must be caught before delivery, not after
  • Multimodal inputs must be handled without losing information during format conversion
  • The pipeline must be traceable at each stage — not just at the final output

These requirements ruled out a single-prompt architecture from the start. They led directly to the hub-and-spoke agent topology.

II

The Hub-and-Spoke Agent Topology

Instead of one large prompt that handles extraction, synthesis, research, critique, and memory all at once, the pipeline uses a hub-and-spoke topology — a design where one central agent (the hub) coordinates several specialised agents (the spokes), each with a single clearly defined responsibility. A failure in the Research Agent does not affect the Extraction Agent's output. A Critic failure triggers a targeted regeneration cycle — not a full pipeline restart.

Extraction Agent — Multimodal Input Normalisation

The Extraction Agent converts all input formats into a single unified text representation before any summarisation begins. It uses OCR (Optical Character Recognition — software that reads text from images or scanned documents) for prescription images and scanned documents. It uses OpenAI Whisper (a speech-to-text model) for audio recordings of consultation notes. It extracts text directly from PDFs and DOCX documents that already have a text layer. By normalising all inputs in this dedicated stage, the Summary Agent always receives clean, uniform text — it never sees raw image files, partially parsed PDFs, or unprocessed audio.

Research Agent — Conditional External Lookup

The Research Agent does not run on every visit — only when the visit context contains signals that need external verification. These signals include potential drug interactions, unfamiliar medication names, dosage ranges outside common clinical norms, or guidelines that require up-to-date evidence. The agent queries external clinical sources via the Brave Search MCP (Model Context Protocol — a standard interface that controls which tools the AI can call and how) server. Results are injected into the Summary Agent's generation step as clearly labelled external references — not as direct clinical claims. This prevents the research step from introducing new hallucinations while still adding relevant context.

Summary Agent — Orchestration Hub

The Summary Agent builds a unified visit context from all extracted inputs and any research evidence the Research Agent gathered. It then generates the structured 3-section HTML summary: chief complaint and assessment, clinical findings and observations, treatment plan and follow-up. It acts as the hub — coordinating fixed handoffs to the Critic Agent for quality review and the Evidence Agent for source mapping. The handoffs are deterministic, not the result of an open-ended agent loop.

Critic Agent — Pre-Delivery QA Gate

The Critic Agent reviews every generated summary before it reaches the clinician. It checks for four types of failures:

  • Factual contradictions — claims that directly conflict with the extracted source input
  • Clinical omissions — findings present in the source that are missing from the summary
  • Hallucinated specifics — names, dosages, dates, or measurements with no source support
  • Structural violations — deviations from the expected 3-section format

If the summary fails on any category, the system runs a tournament of up to 3 regenerated candidates. The Critic's failure analysis is injected as negative constraints into each new attempt, and the best-scoring output is selected. This adds latency but ensures no unverified clinical claims reach the clinician.

III

The Integrity Layer: Evidence Linking & Format Guardrails

The Critic Agent catches hallucinations before delivery. Evidence Linking makes every surviving claim traceable after delivery. Together with Persistent Format Guardrails, these three mechanisms form a layered correctness contract — no single mechanism is enough for clinical-grade output quality. All three work together to close different types of error.

Evidence Agent — Source-to-Claim Mapping

Every statement in the finalised summary is mapped back to the specific source excerpt that supports it. The Evidence Agent processes the finalised summary and the original extracted text, producing a claim-to-source index. The clinician can click any claim in the UI to see the exact input excerpt it came from. In a clinical context, the ability to trace and verify every claim is a safety requirement. If a statement cannot be traced to a source, it is flagged as unsupported and shown as an evidence gap — not silently included in the final output.

Persistent Format Guardrails

When the Critic Agent finds format or structural violations in a summary — missing section headers, wrong subsection order, malformed HTML, clinical information in the wrong section — those violations are saved as format guardrails for that doctor's profile in the database. Future generation prompts for the same doctor include these accumulated guardrails as explicit negative constraints in the system prompt. Output quality improves across sessions without any manual prompt engineering. The pipeline learns from its own Critic failures and stores that learning as doctor-specific generation rules.

Zero Hallucination Contract

The combination of Critic QA, Evidence Linking, and Persistent Guardrails forms a layered zero hallucination contract. The Critic catches errors in the current run. Evidence Linking makes any errors that survive the Critic visible to the clinician before they act on the summary. Persistent Guardrails reduce the chance of the same type of error appearing in future runs. No single layer is enough — all three must work together in sequence.

IV

Longitudinal Memory: RAG for Patient History

A clinician seeing a returning patient should not need to re-enter history that is already documented. The longitudinal memory layer makes prior visit context automatically available to the Summary Agent. The clinician does not need to find and re-attach previous summaries. And there is no privacy risk of mixing in history from other patients.

DynamoDB for Cross-Visit Context

After each visit, the finalised summary is converted into an embedding (a numerical representation of text meaning) and stored in a DynamoDB table keyed by patient ID. On the next visit for the same patient, the system uses RAG (Retrieval-Augmented Generation — a technique where the AI searches stored information before generating a response) to find semantically similar context from previous summaries. Prior diagnoses, known allergies, current medication history, and documented observations are injected as background context into the Summary Agent's generation step. Retrieval is strictly scoped to the current patient — cross-patient retrieval is architecturally impossible because DynamoDB queries are keyed by patient ID at the query layer.

Scoped Chat Assistant

The Chat Agent gives the clinician a conversational interface to ask questions about the current patient's history and visit context. It has read access only to the current patient's visit summaries and consultation context. Cross-patient data access is not just prohibited by policy — it is architecturally impossible. The chat session starts with a patient-scoped context object that is the only data source available to the agent. Clinical answers are always grounded in documented evidence. The Chat Agent cannot speculate beyond what is in the patient's record.

V

Real-Time Pipeline Streaming

The full pipeline — extraction, optional research, summarisation, Critic QA, evidence linking, and memory indexing — takes 30 to 60 seconds under typical load. Making the UI wait silently until completion is not acceptable. The clinician has no visibility into what the pipeline is doing, and loses all progress if the connection drops during the wait.

SSE-Driven Incremental Updates

Each pipeline stage sends an SSE (Server-Sent Events — a technology that lets the server push live updates to the browser without the browser asking repeatedly) event as it finishes: extraction_complete, research_done, summary_draft_ready, critic_pass_complete, evidence_linked, memory_indexed. The frontend updates incrementally as each stage completes, giving the clinician real-time visibility into pipeline progress. If the Critic triggers a regeneration tournament, a critic_regenerating event is sent so the clinician knows why the pipeline is taking longer. The UI is never stuck in an unclear waiting state.

Resumable SSE Jobs

Job state is saved in Upstash Redis keyed by job_id. If the client disconnects mid-pipeline — browser tab closed, network interruption, mobile app in the background — the pipeline keeps running on the server. The client reconnects with the job_id and resumes streaming from the last completed stage. No work is re-executed and no stage outputs are lost. The Redis job record tracks the completion state of each pipeline stage, so the resumption point is exact — the client only receives the events it missed, not a full replay from the beginning.

VI

Operations

FastAPI (Python)

Pipeline orchestration, SSE streaming host, multimodal file handling, and job state management. Handles agent handoff coordination between pipeline stages and exposes the SSE endpoint for incremental progress updates.

Next.js / React + TypeScript

SSE event consumption and incremental stage rendering — each stage event triggers a UI update. Evidence linking UI lets clinicians expand any summary claim to inspect its source excerpt. Scoped chat interface per patient session.

OpenAI Agents SDK

Hub-and-spoke orchestration. Handoffs between Summary, Extraction, Research, Critic, and Evidence agents via structured handoff primitives. Bounded agent runs prevent open-ended loops in the Critic regeneration tournament.

OpenAI Whisper

Audio transcription for voice-recorded consultation notes. Runs within the Extraction Agent before any summarization; output normalized to the same text format as all other inputs so the Summary Agent sees a uniform context.

DynamoDB (RAG)

Longitudinal patient memory via embedding and similarity retrieval. Summaries indexed by patient ID. Cross-patient retrieval is architecturally prevented at the query layer — not just by policy.

Upstash Redis

Job state persistence for resumable SSE sessions. Each pipeline job stored as a Redis hash keyed by job_id with per-stage completion flags. HTTP-based access compatible with App Runner without persistent TCP connections.

MCP (Brave Search)

Research Agent's external lookup mechanism for clinical guidelines and drug interaction data. All external queries go through the typed tool contract; tool call logs captured per pipeline run for auditability.

Resend

Email delivery for summary dispatch. Verified-send pattern: server confirms send receipt before recording delivery success. Failed sends surface as delivery errors in the job state — not silent failures.

Clerk

Authentication and user identity. Scopes all patient data access by authenticated user. JWTs verified on every API request. Patient DynamoDB queries and job history queries both scoped to the verified user identity before any data operation.

Docker + AWS App Runner + ECR

Containerized deployment with automatic HTTPS, health check routing, and request-based scaling. App Runner provides the persistent compute environment needed for multi-stage pipeline jobs that exceed Lambda timeout limits.
