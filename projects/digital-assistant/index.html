Case Study · 01

Digital Assistant

Agentic Systems Engineering on AWS Serverless

Building a production-grade agentic assistant means solving for execution durability, not just capability. This case study covers the asynchronous (running in the background without blocking other tasks) architecture that survives serverless timeouts, the truth-gating protocol that eliminates hallucinated action claims, and the Compare-And-Swap (CAS) — a method that prevents two processes from updating the same data at the same time — quota governance that keeps the system cost-bounded at scale.

Architecture Diagram

I

The Strategic Context: Navigating Cloud Physics

Most agentic chat applications fail in production because they treat AI tool-calling as a simple request-response loop. Real-world agentic workflows — recursive web research, multi-page document parsing, multi-step artifact generation — are unpredictable by nature and high-latency.

In a serverless (cloud environment where the infrastructure is managed automatically and code runs only when needed) AWS environment, these tasks frequently hit a hard infrastructure limit: the 29-second timeout of API Gateway (the AWS service that routes incoming requests to the backend). Running agentic processes synchronously leads to Zombie Processes — the backend keeps computing while the user's connection is cut, resulting in a total loss of session state and wasted compute costs. The user sees nothing; the work is gone.

I built this system to solve for Execution Durability — making sure agentic work survives cloud volatility by aligning the architecture with the actual constraints of serverless compute, not fighting against them.

II

The Architecture: Async Job Orchestration & Hybrid Routing

To solve for reliability, I built a Hybrid Risk Routing architecture that separates the user's perceived latency from the system's actual execution time.

Risk-Tiered Routing

The system classifies incoming requests at the entry point. Low-risk conversational messages follow a guarded synchronous path for sub-second responses. High-risk or tool-dependent requests — PDF generation, email delivery, deep web research — are automatically routed to the Async Job Model. No manual configuration from the user is needed.

The Orchestrator — API Lambda

The FastAPI control plane receives the user request and adds it to a job queue in Upstash Redis (an in-memory data store used as a job queue). It returns a persistent job_id immediately — well within the API Gateway timeout window. The client never waits for tool execution. It receives a job handle and continues. This single design decision makes the entire architecture timeout-proof.

The Decoupled Worker — Worker Lambda

A separate AWS Lambda worker function takes full responsibility for the execution lifecycle. It picks up the job from the queue and runs the MCP (Model Context Protocol — a standard interface that controls which tools the AI can call and how) -orchestrated tool chain:

  • Brave Search for web research
  • WeasyPrint for PDF generation
  • Resend for email delivery
  • File reader for PDF, DOCX, and plain text ingestion
  • Memory MCP server for per-user knowledge extraction

The worker runs entirely outside the API Gateway's timeout domain. It can run as long as the work requires, saving output files to S3 and progress state to Redis.

Execution Durability & Client Resync

The API + Worker pattern means the agent keeps working even if the client session is interrupted or the browser is closed. The client resyncs seamlessly on reconnection, pulling completed results without re-running expensive LLM tool chains. There is no lost state, no re-execution cost, and no wasted compute.

III

The Integrity Layer: Truth-Gated Canonical Rendering

Action Hallucination happens when an LLM claims it completed a task — created a download link, sent an email — but never actually did it. To prevent this, I built the Truth-Gated Canonical Rendering protocol. It strictly separates what the AI intended from what the system verified. The result is zero placeholder links and zero unverified action claims in the final output.

Stage 1 — Approved-Memory Validation

If the user has saved approved memory items, a separate validator model checks whether the assistant's response follows those items. If not, the system reruns the response with targeted fix instructions. Approved memory is not just added to the prompt and hoped for — it is backed by an explicit check after the response is generated.

Stage 2 — Targeted PDF Error Recovery

If a generate_pdf_from_text tool call fails with a PDF_INPUT_INVALID code, the backend triggers one controlled retry. The retry fixes only the formatting or JSON structure, not the content. If that retry also fails, the system returns a clear error message — it never pretends the PDF exists.

Stage 3 — Required-Tool Enforcement

The backend checks the list of tool events and confirms that each required tool actually ran, in the correct order. This prevents "execution bluffing." If the user asked for a PDF and email send but the model only wrote text about those actions without calling the tools, the response is rejected.

Stage 4 — Truth-Context Normalization

Raw tool outputs come in many formats — nested JSON, wrapper objects, inline strings, MCP response structures. The system unwraps and normalises all of them into three clear types of evidence: artifacts (generated PDFs with verified download URLs), action outcomes (email delivery status with email IDs), and search results (source URLs with tracking parameters removed).

Stage 5 — Canonical Rendering

The final response shown to the user is built using only verified evidence. If a PDF exists, the system attaches the verified download URL. If an email was sent, the system confirms with the email ID. Any links the model mentions that are not in the verified evidence set are removed. Source lists come from normalised search results, not from the model's raw text.

Stage 6 — Truth-Gate Validation

The rendered response must not claim more than the tool evidence supports. If the output contains unverifiable links, cites sources the search tool never returned, or claims an action result that does not match the recorded tool outcome, the truth gate blocks it. The AI cannot lie about what it actually did.

Stage 7 — Truth-Fix Loop

If the truth gate blocks the output and the issue is automatically fixable, the system builds targeted fix instructions and reruns the provider. The new output is merged back into the verified evidence context and re-rendered. This loop is limited by TRUTH_FIX_MAX_ATTEMPTS. It is designed to fix phrasing or unsupported claims — not to trigger open-ended regeneration.

IV

Operations: Observability & Quota Hardening

Production reliability needs both cost controls and visibility into what the system is doing. The system uses OpenTelemetry (OTel) — a standard toolkit for tracking requests as they move through a system — for tracing every job across both the API and Worker Lambda functions. Every job has a traceable path from when it was added to the queue through to final output delivery.

CAS-Style Quota Governance

Daily limits — token budget, PDF exports, email sends — are enforced using Compare-And-Swap (CAS), a method that prevents two processes from updating the same data at the same time. This stops double-spending in situations where multiple Lambda functions run at once and try to use the same quota slot. The system stays cost-aware and protected against runaway loops, even under heavy load. Live quota counters are shown directly in the UI — no separate settings page needed.

User-Approved Memory

When the system extracts memory candidates from a conversation, it shows them to the user in a dedicated Memory tab. The user approves or rejects each one — nothing is applied automatically. Only approved memories are injected into the system prompt at the highest priority. This keeps the user in control and prevents incorrect or misinterpreted context from mixing into future responses.

Immutable Infrastructure & Credential-Less CI/CD

The entire infrastructure is defined as code using Terraform — Lambda, API Gateway, S3, CloudFront, ECR, IAM roles, and Secrets Manager references. No manual configuration in the AWS console. Deployments run through GitHub Actions with OIDC (OpenID Connect — a security standard that lets services verify identity without storing long-term credentials) federation. No long-lived AWS access keys exist anywhere in the pipeline. On every push to main, the Docker image is built, pushed to ECR, and Lambda functions are updated.

Tech Stack with Reasoning

FastAPI on AWS Lambda

Container images via ECR. Lambda for zero server management and cost efficiency; FastAPI for clean async patterns and built-in OpenAPI documentation.

Next.js (static export) + S3 + CloudFront

Frontend deployed as static files to S3, served globally via CloudFront CDN. No SSR costs; CloudFront handles HTTPS termination, custom domain, and edge caching.

API Gateway (REST)

Entry point for all API traffic. The 29-second timeout here is the constraint that necessitated the entire async architecture — treating infrastructure limits as first-class design inputs.

Upstash Redis

Serverless Redis for the async job queue and CAS-style quota counters. Zero infrastructure overhead; HTTP-based access compatible with Lambda's ephemeral, stateless execution model.

S3 (Memory + Artifacts)

Conversation history (5 most recent sessions, 50-message context window), approved memory files, and generated artifacts stored as structured JSON in S3. Lazy expiration of approved memory on read. Durable and cheap — no database needed for conversation persistence.

MCP Tool Servers (stdio)

Three server families: Brave Search for web research, a core action server (WeasyPrint PDF, Resend email, file reader), and a separate Memory MCP server for knowledge extraction. Each isolated behind a subprocess boundary with AsyncExitStack lifecycle management. Memory extraction runs async in API mode, sync in worker mode. All subprocesses inherit trace context for coherent OpenTelemetry spans.

Multi-Provider LLM (Grok + AWS Bedrock)

Two distinct execution paths: Grok uses the openai-agents runtime with delegated tool decision-making; Bedrock uses a custom tool loop around the converse API with explicit toolChoice forcing for required tools. Region-prefixed model ID derivation (apac, us, eu) handles Bedrock's region-specific access patterns. Provider routing ensures resilience without surfacing failures to the user.

OpenTelemetry → Sentry

Distributed traces span both Lambda functions with full context propagation. Error boundaries captured at the tool level — not just at the Lambda boundary — for precise diagnosis.

Terraform

All infrastructure as code: Lambda, API Gateway, S3, CloudFront, ECR, IAM roles, Secrets Manager references. No manual console configuration; full environment reproducibility.

GitHub Actions (OIDC)

CI/CD with federated identity — no long-lived AWS credentials in the pipeline. OIDC trust means the pipeline assumes an IAM role per-run with minimum required permissions.
