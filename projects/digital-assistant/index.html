Digital Assistant

Agentic Systems Engineering on AWS Serverless

Engineering a production-grade agentic assistant requires solving for execution durability, not just capability. This case study covers the async architecture that survives serverless timeouts, the truth-gating protocol that eliminates hallucinated action claims, and the CAS-style quota governance that keeps the system cost-bounded at scale.

I

The Strategic Context: Navigating Cloud Physics

Most agentic chat applications fail in production because they treat AI tool-calling as a simple request-response loop. Real-world agentic workflows — recursive web research, multi-page document parsing, multi-step artifact generation — are inherently non-deterministic and high-latency.

In an AWS serverless environment, these tasks frequently collide with a physical infrastructure limit: the 29-second hard-timeout boundary of API Gateway. Attempting to run agentic processes synchronously leads to Zombie Processes — the backend continues to compute while the user's connection is severed, resulting in a total loss of session state and wasted compute costs. The user sees nothing; the work is gone.

I engineered this system to solve for Execution Durability — ensuring that agentic work survives the volatility of cloud environments by aligning the architecture with the actual physics of serverless compute, not fighting against it.

II

The Architecture: Async Job Orchestration & Hybrid Routing

To solve for reliability, I implemented a Hybrid Risk Routing architecture that decouples the user's perceived latency from the system's execution physics.

Risk-Tiered Routing

The system classifies incoming intents at the entry point. Low-risk conversational prose follows a guarded synchronous path for sub-second responses. High-risk or tool-dependent requests — PDF generation, email delivery, deep web research — are automatically routed to the Async Job Model without any manual configuration from the user.

The Orchestrator — API Lambda

The FastAPI control plane receives the user intent, enqueues it into Upstash Redis, and returns a persistent job_id immediately — well within the API Gateway timeout window. The client never blocks on tool execution; it receives a job handle and proceeds. This single design decision is what makes the entire architecture timeout-proof.

The Decoupled Worker — Worker Lambda

A separate AWS Lambda worker function assumes full responsibility for the entire execution lifecycle. It dequeues the job and runs the MCP-orchestrated tool chain:

  • Brave Search for web research
  • WeasyPrint for PDF generation
  • Resend for email delivery
  • File ingestion and memory extraction

The worker operates entirely outside the API Gateway's timeout domain — it can run for as long as the work requires, persisting output artifacts to S3 and progress state to Redis.

Execution Durability & Client Resync

This API + Worker pattern ensures that even if the client session is interrupted or the browser is closed, the agent continues its work uninterrupted. The client resyncs seamlessly upon reconnection, pulling completed results without re-running expensive LLM tool-chains. There is no lost state, no re-execution cost, no wasted compute.

III

The Integrity Layer: Truth-Gated Canonical Rendering

To eliminate Action Hallucination — where an LLM claims a successful outcome (a generated download link, a sent email confirmation) that was never actually verified — I engineered the Truth-Gated Canonical Rendering protocol. This contract-first architecture strictly separates Intent from Fact, ensuring a 0% rate of placeholder links and zero unverified action claims in the final output.

Step 1 — Intent Contract

The LLM produces a strict IntentEnvelope defining the requested action type and parameters. It is never permitted to draft the final delivery prose or provide its own links. Its role ends at declaring what it wants to happen — not at claiming that it happened. This boundary is enforced at the API layer, not through prompting alone.

Step 2 — Canonical Execution

The server independently executes the tool — calling the Resend API, running the WeasyPrint PDF engine, or performing the Brave Search — and captures a verified TruthContext containing the actual file path, delivery confirmation status, or structured result data. The LLM has no visibility into this execution step.

Step 3 — Deterministic Rendering

The final user-facing response is generated via server-side templates using only the verified TruthContext. If the tool fails, the Truth Gate blocks the success claim entirely and returns a structured error — the AI cannot lie about its own progress. The result is a system where every link in the UI is a real, verified artifact path, and every action confirmation reflects a real execution result.

IV

Operations: Observability & Quota Hardening

Production reliability is governed by granular cost controls and deep observability. The system is instrumented with OpenTelemetry (OTel) for trace-level job correlation across both the API and Worker Lambda boundaries — every job carries a traceable span from enqueue through final artifact delivery.

CAS-Style Quota Governance

Daily limits — token budget, PDF exports, email sends — are enforced using Compare-And-Swap (CAS) optimistic concurrency in Upstash Redis. This prevents double-spend in distributed environments where multiple concurrent Lambda invocations could race to consume the same quota slot. The system remains cost-aware and protected against runaway agentic loops, even under concurrent load. Real-time counters surface live quota state in the UI without a separate settings page.

User-Approved Memory

Memory candidates extracted from conversations are surfaced to the user in a dedicated Memory tab for explicit approval or rejection — they are never automatically applied. Only approved memories are injected into the system prompt at the highest priority tier. This maintains user trust and eliminates the risk of contamination from incorrectly extracted or misinterpreted context.

Immutable Infrastructure & Credential-Less CI/CD

The entire infrastructure is provisioned as code via Terraform — Lambda, API Gateway, S3, CloudFront, ECR, IAM, and Secrets Manager references, with no manual console configuration. Deployments run through GitHub Actions with OIDC federation — no long-lived AWS access keys exist anywhere in the pipeline. On every push to main: the Docker image is built, pushed to ECR, and Lambda functions are updated atomically.

Tech Stack with Reasoning

FastAPI on AWS Lambda

Container images via ECR. Lambda for zero server management and cost efficiency; FastAPI for clean async patterns and built-in OpenAPI documentation.

Next.js (static export) + S3 + CloudFront

Frontend deployed as static files to S3, served globally via CloudFront CDN. No SSR costs; CloudFront handles HTTPS termination, custom domain, and edge caching.

API Gateway (REST)

Entry point for all API traffic. The 29-second timeout here is the constraint that necessitated the entire async architecture — treating infrastructure limits as first-class design inputs.

Upstash Redis

Serverless Redis for the async job queue and CAS-style quota counters. Zero infrastructure overhead; HTTP-based access compatible with Lambda's ephemeral, stateless execution model.

S3 (Memory + Artifacts)

Conversation history, approved memory files, and generated artifacts stored as structured JSON in S3. Durable and cheap — no database needed for conversation persistence.

MCP Tool Servers (stdio)

Brave Search, WeasyPrint (PDF), Resend (email), file reader, memory extractor — each isolated behind an MCP server boundary. Independently testable and swappable without modifying the agent loop.

Multi-Provider LLM (Grok + AWS Bedrock)

Primary and fallback LLM providers. Multi-provider routing ensures resilience if one endpoint is degraded or rate-limited, without surfacing provider failures to the user.

OpenTelemetry → Sentry

Distributed traces span both Lambda functions with full context propagation. Error boundaries captured at the tool level — not just at the Lambda boundary — for precise diagnosis.

Terraform

All infrastructure as code: Lambda, API Gateway, S3, CloudFront, ECR, IAM roles, Secrets Manager references. No manual console configuration; full environment reproducibility.

GitHub Actions (OIDC)

CI/CD with federated identity — no long-lived AWS credentials in the pipeline. OIDC trust means the pipeline assumes an IAM role per-run with minimum required permissions.
