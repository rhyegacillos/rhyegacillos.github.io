Case Study · 02

IdeaGen

Multi-Model Decision Intelligence Platform

Building a multi-model decision workflow means solving for output comparability, not just generating more text. This case study covers the contract-first validation architecture that enforces structured outputs across four LLM (Large Language Model — the AI brain that reads text and generates responses, like GPT-4 or Gemini) providers, the deterministic (always producing the same result for the same input — no randomness) financial modelling layer that prevents hallucinated investment projections, and the artifact lifecycle that makes every analysis persistent and reusable.

Architecture Diagram

I

The Strategic Context: Why Single-Model Ideation Fails

A single AI model gives one perspective. That perspective is shaped by one provider's training data, RLHF (Reinforcement Learning from Human Feedback — a training method where human preferences shape the AI's output style) preferences, and capabilities. When the AI is recommending whether to invest capital or pursue a strategy, relying on one perspective is a real risk. Different leading models reason differently:

  • OpenAI tends toward structured decomposition
  • Gemini toward pattern synthesis
  • DeepSeek toward cost-efficient reasoning
  • Grok toward real-time-aware analysis

Sending the same ideation task to all four models and scoring each output against a structured rubric gives something no single-model system can provide — a cross-model consensus signal that also shows where models disagree. The engineering challenge is making those outputs comparable, traceable, and ready for a decision — not just a collection of text. That requires enforcing a fixed output structure so every model produces the same shape of data, computing financial numbers programmatically so they are never hallucinated, and storing every analysis so it can be retrieved and reused weeks later.

II

The Architecture: Parallel Multi-Model Orchestration

The generation layer is built around three connected properties. First, every output has a guaranteed structure. Second, every provider failure is automatically recovered within its own slot. Third, multiple models generate against the same prompt at the same time so their outputs can be ranked side by side. These three properties depend on each other — the fixed structure is what makes parallel scoring possible, and the per-provider fallback chain is what keeps each slot reliable when real-world providers go down.

Contract-First Structured Outputs

Every LLM generation step produces a typed response that is validated against a Pydantic JSON schema (a Python tool that checks whether data has the correct format and field types). If validation fails, the system retries with a correction prompt that includes the specific violations from the failed response — field names, type mismatches, missing required properties. After 3 failed attempts, the system falls back to a deterministic safe payload that satisfies the schema with explicit null values and error annotations. The frontend always receives a predictable structure regardless of LLM behaviour — no UI crashes, no partial renders, no silent missing fields.

Per-Provider Model Fallback

Each provider slot has its own internal model chain. If the primary model hits a temporary failure or rate limit, the system falls back to a less powerful model from the same provider — not a different provider. OpenAI falls back within OpenAI; Gemini falls back within Gemini. Each parallel generation task is independently resilient without cross-provider routing. The provider and model that fulfilled each slot are recorded in the artifact metadata so you can see exactly which model produced which output.

Parallel Multi-Model Generation

Up to 5 model configurations generate outputs at the same time for the same ideation prompt using asyncio.gather (a Python tool for running multiple tasks concurrently). Each output is scored independently against the same rubric — novelty, feasibility, market size, competitive moat, execution risk — producing numerical rankings for direct head-to-head comparison. Because every output is validated against the schema before scoring, the evaluator always receives clean, comparable inputs regardless of which model produced them. Uniformity at the generation stage is what makes comparability at the scoring stage possible.

III

The Integrity Layer: Deterministic Financial Modeling

The most dangerous failure mode in an AI investment platform is not a bad recommendation — it is a confidently stated hallucinated number. An LLM can generate financial projections that look precise: "Year 3 revenue: $4.2M, CAGR (Compound Annual Growth Rate — the average yearly growth rate over a period): 34%." These numbers have no real basis. They are pattern-matched outputs that look like financial analysis but are not computed from any formula. In a decision context, that is the difference between a useful tool and a dangerous one.

The Execution Plan module solves this with a strict separation between what the LLM is allowed to produce and what the system computes programmatically. This boundary is enforced at the API layer — it is not a suggestion in a prompt, it is an architectural constraint.

Separation of Computation from Narration

All financial projections in Execution Plans are calculated by the system using explicit formulas — revenue by growth rate and market penetration, cost structure by category ratios, break-even by contribution margin, sensitivity ranges by bounded parameter variation. The LLM only contributes narrative framing and assumption justification. It explains what the numbers mean, interprets the sensitivity analysis, and describes the strategic reasoning behind the assumptions. It does not produce any numbers. The generation prompt explicitly forbids numerical financial claims.

Go / Conditional / No-Go Decision Gates

The execution plan includes a deterministic gate evaluation. If IRR (Internal Rate of Return — a financial metric showing the expected annual growth rate of an investment) falls below a configured threshold, or break-even extends beyond a set number of years, the gate returns conditional or no-go with a clear reason for each threshold violation. Profitability recovery logic then identifies which assumptions would change a no-go to a conditional — for example, reducing initial headcount by 20% or extending the runway by 6 months — and shows these as named levers. Sensitivity testing also calculates which input variables have the biggest impact on the final gate outcome, giving decision-makers a ranked list of what to watch and negotiate.

Audit Trail by Construction

Because projections are calculated from explicit inputs and formulas that are saved alongside the artifact in PostgreSQL, every number in the output can be re-derived at any point after generation. There is no "the model said so" — every projection has a traceable source formula. This makes the output defensible in a due-diligence context. Any stakeholder can inspect the assumption inputs, verify the formula, and reproduce the result independently — without access to the original LLM session.

IV

The Artifact Lifecycle: Persistent Decision Intelligence

Most AI tools produce temporary outputs — close the tab and the work is gone. IdeaGen treats every output as a first-class persistent artifact with its own lifecycle state, re-export options, and the ability to feed into later workflow steps. A run generated today can become the input for an execution plan next week — no manual copy-paste needed. The artifact graph manages the dependency chain.

Artifact Persistence Model

Every run, comparison, decision summary, and execution plan is saved to PostgreSQL with its own lifecycle state: draft → finalized → archived. Artifacts can be retrieved at any stage, re-exported to PDF via WeasyPrint, emailed via Resend, or used as inputs for downstream steps. The database model records dependency links explicitly — a Decision Summary references the run IDs it was built from; an Execution Plan references the decision summary it came from. Every artifact has a traceable origin back to the original ideation inputs.

Run-to-Run Comparison

Users select any two saved runs for a side-by-side comparison. The system identifies the winner on each rubric dimension separately, explains the trade-off — where model A scored higher on novelty but lower on execution risk and why — and highlights where the two models contradicted each other on the same concept. This comparison is itself a saved artifact with its own lifecycle state. It can be exported to PDF or used as context for a downstream Decision Summary.

Cache-Before-Infer

Before calling an LLM for scoring or comparison, the system checks whether an identical request already has a cached result. The cache key is a deterministic hash of the input parameters and schema version. This reduces latency and cost for repeated or similar inputs without lowering result quality. Cache entries are scoped to the artifact — when an artifact is re-finalised, its downstream cache entries are invalidated. Stale scoring results are never shown after a source artifact is updated.

V

Operations: App Runner, PostgreSQL, Auth

FastAPI (Python)

Orchestration layer for multi-model routing, retry logic, Pydantic schema validation, PDF generation, email delivery, and deterministic financial computation. Single async service — App Runner handles horizontal scaling.

Next.js + TypeScript

Adaptive frontend with guided flow mode (step-by-step ideation walkthrough) and status flow mode (real-time generation progress). Per-step state managed via localStorage for recovery on page reload. Modal-driven artifact management for retrieving, comparing, and exporting saved runs.

PostgreSQL + SQLAlchemy + Alembic

Persistence for all runs, comparisons, decision summaries, and execution plans. Relational model chosen for complex artifact query patterns and multi-step lifecycle state transitions. Alembic migrations applied automatically on container startup.

WeasyPrint

Server-side HTML→PDF rendering for report exports. Reports use structured HTML templates populated from artifact data — consistent formatting regardless of LLM generation variance.

Resend

Email delivery for PDF report dispatch. Verified-send pattern: server confirms send receipt before recording delivery success in the artifact lifecycle state. Failed sends surface as artifact state errors, not silently dropped.

Clerk

Authentication, session management, and organization scoping. User identity gates all artifact access — no cross-user artifact visibility is architecturally possible. JWTs verified on every API request; all artifact queries scoped to the verified user ID.

Docker + AWS App Runner + ECR

Container-based deployment. App Runner handles automatic HTTPS, request-based scaling, and health check routing with no manual server management. ECR stores versioned images; App Runner pulls the latest on each deploy.

AWS RDS (PostgreSQL)

Managed database with automated backups and point-in-time recovery. Chosen over self-hosted PostgreSQL for durability guarantees and operational simplicity. Connection pooling via SQLAlchemy's async engine.

AWS Secrets Manager

Runtime secrets — API keys, database credentials, and third-party service tokens — stored in Secrets Manager and injected into the App Runner container at startup. No secrets in source control or environment file artifacts.

Route 53

ALIAS record maps the custom domain to the App Runner service URL. Certificate provisioning handled by App Runner's managed TLS. DNS health check failover configured for additional resilience during service restarts.
