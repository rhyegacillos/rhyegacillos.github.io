Digital Assistant — Case Study

Persona-driven, serverless assistant with contract-first truth gating and safe tool orchestration.

Role

Built orchestrator, tool integrations, async-worker pipeline, and truth-gating flow.

Problem

Users need an assistant that can call real tools (search, generate PDFs, send email) without producing unverifiable claims or performing unsafe actions. Tool-driven workflows are high-risk and require validation before delivery.

Approach

We classify each request by risk: low-risk chat remains conversational; high-risk actions (PDF/email/research) follow a contract-first flow. The LLM returns a strict intent payload, the server executes tools and captures canonical facts (artifact URLs, statuses, sources), and a truth gate validates those facts against the intent. Only when the truth gate passes does the server render and deliver the final output. Long-running jobs run via background workers so the UI remains responsive. Persona behavior and quotas enforce safety and cost controls.

Why this approach

  • Safety: Outputs that claim actions or citations are only published when canonical proof exists.
  • Reliability: Background workers prevent timeouts for multi-step workflows.
  • Control: Quotas and gating limit risky or expensive operations.
  • Auditability: Canonical artifacts and verdicts make outputs traceable and reproducible.

Tech

FastAPI · Next.js/React · TypeScript · AWS Lambda/API Gateway · S3/CloudFront · Terraform · Redis/Upstash · OpenTelemetry → Sentry

Outcome

Delivered a production-ready persona agent that safely executes and validates high-risk tool workflows (PDF/email/research) with server-side quota governance and observability. See architecture and one-pager for implementation details.
